# 为什么这个方案更好：开销降低的核心逻辑

## 先看全量重建的代价

每次更新后重建的步骤是：

```
1. 枚举所有元路径 P （最多 O(|𝒫|) 个）
2. 对每个 P，重新枚举所有 path instances → O(|V_C|² × L)
3. 对每个 G_P，从零做 k-core 分解 → O(|E_P|)
4. 重建整棵 CM-tree → O(|𝒫| × |V_C|)
```

**代价瓶颈在于：哪怕只改了一条边，也要把所有东西重算一遍。**

---

## 本方案的核心逻辑：最小化"必须做的工作"

### 创新点一的降低开销逻辑

**全量重建：** 枚举所有元路径 × 所有 center pairs × 所有 path instances

**本方案：** 一次 `insertEdge(c, a)` 只改变了 center `c` 经过属性 `a` 的半路径连接能力。

关键观察：

> Star-schema HIN 中，一条 `(c, a)` 边只参与**包含属性类型 `type(a)` 的半路径**，其他半路径完全不受影响。

所以：

| 步骤 | 全量重建 | 本方案 |
|---|---|---|
| 受影响元路径 | 全部 `\|𝒫\|` 个 | 只有 `\|𝒫_aff\|` 个（通常远小于 `\|𝒫\|`） |
| 受影响 center pairs | 所有 `(u,v)` | 只有与 `c` 在 HSI 中有兼容签名的邻居 |
| 需要验证的 path instances | 全量枚举 | 只验证签名兼容的候选对，再用 support count 过滤 |

**关键降幅：** path instance 枚举规模从 `O(|V_C|²)` 降到 `O(deg(c)^L)`，其中 `L` 是半路径长度，`deg(c)` 是 center 的局部邻居数。

---

### 创新点二的降低开销逻辑

#### (k,P)-core 维护部分

**全量重建：** 对每个 `G_P` 从零做 BFS/DFS 剥离，复杂度 `O(|E_P|)`

**本方案 Batch Order-P：**

order-based 方法的核心性质是：

> **core number 变化具有局部传播性。** 一条边的插入/删除，只能影响与该边相邻的"局部 shell 闭包"内的顶点。

设一次原子更新产生 `|ΔE_P|` 条 projected-edge 变化，受影响顶点闭包大小为 `|V_P^aff|`，则：

| 步骤 | 全量重建 | 本方案 |
|---|---|---|
| 需要处理的顶点 | 所有 `\|V_C\|` | 只有 `\|V_P^aff\|`（局部闭包） |
| 复杂度 | `O(\|E_P\|)` per P | `O(\|ΔE_P\| × shell_depth)` per P |

在真实图中，一次局部更新影响的 shell 深度往往很浅（通常不超过 2～3 层），所以实际加速效果非常显著。

#### CM-tree 修复部分

**全量重建：** 重建整棵树，复杂度 `O(|𝒫| × |V_C|)`

**本方案 Delta-CM：**

CM-tree 的结构性质是：

> **嵌套单调性：** 较长元路径的 core 结果一定是较短路径 core 结果的子集。

利用这个性质：

- 父节点结果未变 → 子树全部剪枝，无需修复
- 某节点社区变空 → 整个子树直接废弃
- 边界顶点集不变 → 下游查询结果直接复用

| 修复代价 | 全量重建 | Delta-CM |
|---|---|---|
| 必须访问的节点 | 整棵树 `\|𝒫\|` 个节点 | 只有 dirty subtree，通常 `\|𝒫_aff\|` 个 |
| 查询结果重算量 | 全部 | 只有 dirty + 无法剪枝的节点 |

---

## 总结对比

| 阶段 | 全量重建 | 本方案 | 降低原因 |
|---|---|---|---|
| 受影响路径定位 | 枚举所有 P | 只看 `\|𝒫_aff\|` | Star-schema 的局部性：一条边只影响含特定属性类型的路径 |
| path instance 变化 | 全量枚举 | 签名过滤 + support count | 相同签名才可能形成 path instance，其余直接跳过 |
| core 维护 | 全图 core 分解 | 局部 shell 闭包内传播 | core number 变化有局部传播性，不扩散到整图 |
| CM-tree 修复 | 重建整棵树 | 只修 dirty 子树 | 嵌套单调性让大量节点可以直接剪枝 |

**本方案更好的本质原因是：它利用了 Star-schema HIN 结构的三个内在性质来精确定位和最小化需要重算的范围，而不是无视结构做暴力重算：**

1. **Star-schema 局部属性连接性** → 一条边只影响含特定属性类型的路径（创新点一）
2. **core number 变化的局部传播性** → 不扩散到整图（创新点二 core 维护）
3. **CM-tree 的嵌套单调性** → 大量节点可直接剪枝（创新点二索引修复）
